# 결정 기록

> 왜 그렇게 정했는지를 남긴다. 무엇을 정했는지가 아니라 **왜**.

## 2026-04-27 — 프로젝트 시작

### D-001. 1단계는 웹 없이 Typst R&D만

**결정**: 회원가입·결제·인쇄 연동·웹 UI 모두 1단계에서 제외.

**이유**:
- 핵심 가설은 "콘텐츠 JSON이 깨끗한 PDF로 컴파일되는가" 단 하나.
- 웹/DB 먼저 만들면 가설 검증이 늦어지고, 가설 실패 시 매몰비용만 커진다.
- PDF 품질이 안 나오면 그 위에 뭘 올려도 의미 없음.

**대안과 기각 사유**:
- 웹 + Typst 동시 진행 → 컴파일 성공 후 정렬·금칙 미세조정에 시간 다 쓸 가능성.
- LaTeX → 한글 처리 의존성 무겁고, 모던 워크플로 아님.
- IDML/InDesign 자동화 → 라이선스 + 파이프라인 복잡도 폭증.

### D-002. AI는 핵심 경로에 두지 않는다

**결정**: 조판 파이프라인에서 LLM API 호출 0회.

**이유**:
- 비결정성 = 같은 입력에 다른 출력 = 디버깅 지옥.
- 외부 서버에 원고 보내는 건 프라이버시 리스크.
- 비용/속도/예측가능성/프라이버시를 한 번에 해결.
- AI는 콘텐츠 작성 보조(나중에 사용자 측에서)에만 검토.

### D-003. 스타일 선택 → 콘텐츠 입력 순서

**결정**: 사용자 흐름은 (스타일 미리보기 → 선택 → 콘텐츠 입력).

**이유**:
- 빈 페이지 공포증. 사용자 동기는 "예쁜 디자인 보고 거기 내 글 넣고 싶다"에서 출발.
- 스타일이 1개면 의미 약하지만, 데이터 구조를 처음부터 이렇게 잡아야 확장 가능.

### D-004. 사용자에게 폰트/자간/여백 노출 금지

**결정**: 포맷팅은 단락·제목·강조·인용까지만.

**이유**:
- 자유도 ↑ = 품질 리스크 ↑. 비전문가가 자간 만지면 망친다.
- "디자이너 없이 단행본 만들기"가 본 서비스의 핵심 약속.
- 결과물 일관성으로 브랜드 만들기.

### D-005. format_id = `sinkukpan`

**결정**: 신국판 식별자는 `sinkukpan` (한글 → 학술적 로마자 표기 ALA-LC 기반).

**이유**:
- `b6` 같은 ISO 분류는 일본·국제 출판 분류라 한국 단행본 정체성과 어긋남. 신국판은 152×225로 B6(125×176)와 다름.
- `shinkukpan`은 일본어 '신' 발음(sh-) 영향. 한국어 표기는 'sin-'.
- `sinkukpan`은 ALA-LC/매큔라이샤워에 가까워 학술계에서 통용.
- 미래에 `kukpan`(국판 148×210), `46pan`(46판 127×188) 추가 시 동일 규칙 적용 가능.

### D-006. 콘텐츠 JSON 스키마 v1

**결정**: 다음 구조로 고정. 1단계는 인라인 마크업 없음(plain text only).

```json
{
  "schema": "sungjin-book/v1",
  "meta": { "title": "...", "author": "..." },
  "blocks": [
    { "type": "chapter", "number": 1, "title": "..." },
    { "type": "heading", "level": 2, "text": "..." },
    { "type": "paragraph", "text": "..." },
    { "type": "quote", "text": "...", "by": "..." }
  ]
}
```

**이유**:
- **블록 배열 + type 디스크리미네이터**: 미래 블록 타입(image, table, footnote) 추가 시 호환. 렌더러 중립.
- **schema 버전 필드**: 향후 v2로 마이그레이션 시 분기 가능.
- **인라인 마크업 1단계 제외**: 태평천하 본문은 plain text + 따옴표만으로 충분. 인라인은 v2에서 `runs: [{text, style}]` 도입 예정. 1단계 가설 검증에 불필요.
- **chapter는 별도 블록**: 챕터 분기·홀수페이지 시작 등 조판 로직이 챕터 단위이므로 paragraph와 평면 동거시키면 안 됨.

**기각 대안**:
- Markdown 문자열 1개로 다 표현 → 렌더러별 파싱 차이로 결정론 깨짐.
- 인라인 마크업 즉시 도입 → 1단계 가설(콘텐츠↔PDF)과 무관한 복잡도 증가.

### D-007. separator_id 1단계 기본값 = `plain`

**결정**: 1단계는 챕터 사이 별도 분리자 없이 빈 줄만. id는 `plain`.

**이유**:
- `none`은 부정형이라 미래에 'none + 빈 줄' vs 'none + 아무것도 없음' 같은 모호함.
- `plain`은 중립적 양수 의미. 미래 확장(`stars`, `dingbat`, `flourish`)과 동일 패밀리.

### D-008. 변환 = Typst 네이티브 JSON 로드 (별도 스크립트 0개)

**결정**: `template.typ`에서 `#json("../../../content/foo.json")`으로 직접 읽고 렌더. JSON→Typst 변환 스크립트 안 만든다.

**이유**:
- Typst는 `json()`/`yaml()` 빌트인 지원. 변환 단계 추가 = 의존성+버그 표면 추가.
- 1단계 핵심 가설은 "JSON이 깨끗한 PDF로 컴파일되는가"인데, 중간 변환 스크립트가 끼면 가설 경로가 흐려짐.
- Node/Bun/Python 의존성 0. `typst` 단일 바이너리로 끝.
- 미래에 다른 렌더러(InDesign 자동화 등) 필요해지면 그때 별도 스크립트 추가. JSON 자체는 이미 렌더러 중립.

**기각 대안**:
- TypeScript 변환기 → 의존성·빌드 단계 증가, 1단계엔 오버엔지니어링.
- Python 변환기 → 동일 이유 + 사용자 환경에 Python 강제.

### D-009. 서비스 정체성 — 전자책 + ISBN 무료, 인쇄만 유료 (북극성, 잠정)

**결정**: 사용자가 글을 쓰면 → 전자책 PDF + ISBN까지 **무료**로 제공. 종이책 인쇄(POD)만 유료.

**상태**: **목표(North Star)**. 1단계 검증 + 시장 반응 따라 변경 가능. 확정 아님.

**이유**:
- 진입 장벽 0 — 출판사 등록·디자이너·비용 모두 면제. "디자이너 없이 단행본 만들기" 약속을 끝까지 끌고 감.
- 한 줄 가치 명제 — "무료로 책 만들기"는 흔하지만 **ISBN까지** 무료는 경쟁자 거의 없음.
- 수익은 인쇄로 — sungjinprint(Printology)가 이미 인프라. PDF 만든 작가가 자연스럽게 POD 주문 → 수익 연결.

**메커니즘 (한국)**:
- ISBN 발급 자체는 국립중앙도서관 KOLIS-NET에서 무료.
- 출판사 등록 필요(시·도청 신고제, 거의 무료) → 우리(또는 sungjinprint)가 출판사 자격 보유.
- 납본 의무 — 전자책은 파일 등록으로 갈음.

**다음 모든 결정의 기준점**: UX·기능·과금·기술 선택 시 "이게 무료 약속을 깨지 않나?", "인쇄 전환에 도움 되나?" 두 잣대로 판단.

### D-010. UX = 분할 화면 + 프리셋 (Vellum 노하우 흡수)

**결정**: 들어가자마자 **분할 화면** — 좌(글 쓰는 칸) + 우(PDF 미리보기). 위쪽에 **판형 / 스타일 드롭다운 2개**. 가입 게이트 X.

**이유**:
- Vellum(인디 작가 표준), Notion, Figma, Canva, tldraw 모두 같은 패턴 — 가입 게이트 = 작가 90% 이탈.
- "들어가자마자 즉시 사용" — D-009 무료 약속의 UX 표현.
- 가입은 **다운로드 시점만** (이메일만 받음). 그 전엔 브라우저 localStorage 자동저장.

**카피 톤**: "진짜 무료 책 만들기, 지금 해보세요" / "디자이너 없이도, 책처럼 보입니다".

### D-011. 사용자 결정 자유도 = 판형 + 스타일 (2축)

**결정**: 사용자가 고르는 건 **판형 + 스타일** 2개. 분리자(separator)는 스타일에 흡수.

**이유**:
- Jobs/Rams "less, but better". 결정할 게 적을수록 사용자 진입 빠름.
- 폰트·자간·여백은 시스템 자동 (D-004 연장 — 자유도 = 품질 리스크).
- 코드 내부는 (format, theme, separator) 3축 유지하되, UX는 2축만 노출.

### D-012. 한국 판형 4개 (단계적 추가)

**결정**: 한국 단행본 판형 4종을 단계적으로 지원.

| 판형 | 크기 (mm) | 용도 | 추가 시점 |
|---|---|---|---|
| 신국판 | 152 × 225 | 단행본 표준 | 트랙 A M1 (지금) |
| 국판 | 148 × 210 | 에세이·학술 | 트랙 A M2 |
| 4·6판 | 127 × 188 | 문고·시집 | 트랙 A M3 |
| 크라운판 | 176 × 248 | 실용서·사진책 | 트랙 A M4 |

**이유**: 한국 출판 인기 4종. 더 많으면 사용자 부담(Jobs: 적게).

### D-013. 트랙 모델 — 단계(phase)가 아닌 트랙(track)

**결정**: 작업을 단계(phase)가 아닌 **트랙(track)**으로 본다.

| 트랙 | 성격 | 진척 |
|---|---|---|
| **A. 템플릿 라이브러리** | 계속 자라남(디자이너 작업처럼) | 신국판 클래식 1조합 |
| **B. 웹 시스템** | 점진 발전(한 번 만들면 모든 템플릿 활용) | 0% |
| **C. 운영** (ISBN/납본/POD) | 필요할 때 끼움 | 0% |

**이유**:
- 1단계 PDF는 "끝나는 작업"이 아님 — 스타일·판형·테마 계속 추가됨.
- 트랙 B(웹+미리보기) 먼저 만들면 트랙 A 새 템플릿 검증 시간 1/10.
- "1단계 완성" 표현 → "트랙 A 첫 컷"으로 교체.

### D-014. 기술 스택 (장기 안정 우선)

**결정**: 5년 후 후회 안 할 보수적 선택.

| 영역 | 선택 | 1줄 이유 |
|---|---|---|
| 사이트 뼈대 | **Next.js 15 (App Router)** | 한국 인력풀 풍부, Vercel 무료 |
| 에디터 | **Tiptap v2** | ProseMirror 10년+ 검증, 한국어 사례 풍부 |
| 미리보기 | **Vercel Function + typst CLI 서버 컴파일** | 우리 템플릿 100% 호환, 무료 tier 충당 |
| 인증·DB·스토리지 | **Supabase** | 무료 tier, 한 묶음 |
| UI | **shadcn/ui + Tailwind CSS** | 2026 표준, 자유 커스텀 |
| 호스팅 | **Vercel** | 사용자 이미 보유 계정, Next.js 친화 |
| 결제(미래) | **토스페이먼츠** | 한국 표준 |
| 폰트 | **OFL 5쌍 동봉** (노토 세리프/산스 KR, 나눔명조/고딕, Pretendard) | 재현성 + 상업 이용 자유 |

**기각 대안과 이유**:
- **BlockNote** — Tiptap 위 한 겹 더, 메인테이너 1명. 우리 UX(책 미리보기)에 노션-블록 UX 강점 약함. 우리 커스텀 블록은 결국 Tiptap 레이어.
- **Typst.ts (브라우저 WASM)** — 우리 template.typ가 0.13+ 신기능(`query`, `metadata`, `cjk-latin-spacing`) 사용 중, typst.ts 호환 보장 X. 트래픽 늘 때 R&D로 점진 이전.
- **Astro** — 컨텐츠 사이트엔 좋으나 우리는 에디터 무거운 SaaS, React 생태계가 더 풍부.
- **SvelteKit** — 코드 깔끔하지만 한국 인력 적음.
- **Lexical** — Meta 백킹이지만 한국어 사례 부족, 내부 API 변동 잦음.

**위험 대비책**:
- 미리보기 1-2초 = debounce + 로딩 표시로 흡수.
- 트래픽 1만 명 초과 시 Vercel Pro $20/월 — 그쯤 인쇄 매출이 받침.
