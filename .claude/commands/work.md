# /work — Planner → Generator → Evaluator 하네스 (자가진화형)

> Anthropic $200 하네스 패턴 적용.
> 시작할 때 한 번 실행하면 Plan → Code → Evaluate 전체 흐름이 자동으로 진행된다.
> 핵심 원칙: "만드는 쪽을 비판적으로 만드는 것보다, 별도 평가자를 깐깐하게 튜닝하는 게 훨씬 쉽다."
> Self-Evolving: 매 실행마다 이 파일 자체가 학습하고 개선된다.

## PHASE 0: 진화 + 브랜치 결정 (제일 먼저)

### 0-1. 이 도구 자체 로직 최신화

매 실행마다 이 파일(`work.md`)이 여전히 유효한지 점검한다:

- `CLAUDE.md` 읽기 → 새로 추가된 규칙·아키텍처 변경
- `docs/guardrails/shared/evolution-log.md` → 지난 세션의 `[자동반성]` / `[사용자발견]` 확인
- `git log --oneline -10` → 최근 변경된 도구 설정 확인
- 최근 배운 교훈이 이 파일에 반영되어 있나? 아니면 STEP 5에서 추가할 것

### 0-2. 브랜치 결정 (위험도 평가 필수)

작업 요청을 읽고 **위험도**를 판정:

| 위험도     | 기준                                                                             | 브랜치                                             |
| ---------- | -------------------------------------------------------------------------------- | -------------------------------------------------- |
| **HIGH**   | 다수 파일 리팩토링, 가격/보안 경로 수정, 업로드/인증 시스템 변경, UI 대규모 교체 | **필수 브랜치**: `git checkout -b refactor/{name}` |
| **MEDIUM** | 단일 기능 추가, 1~3개 파일 수정, 새 컴포넌트 신설                                | 권장 브랜치: `feature/{name}`                      |
| **LOW**    | 문구 수정, 단일 버그 한 줄, 주석 정리                                            | main 직접 작업 OK                                  |

**HIGH 작업은 브랜치 없이 진행 금지.** Vercel이 브랜치별 프리뷰 URL을 자동 생성하므로 실측 가능.

### 0-3. 메타 의심

- 이 파일에 **불필요해진 STEP**이 있나? → 제거
- **순서가 비효율적인가**? → 재정렬
- 최근 작업에서 **이 도구가 놓친 패턴**이 있나? → STEP 5에서 이 파일을 직접 수정

## PHASE 1: PLANNER (기획자)

> "무엇을(What)" 만들지 결정한다. "어떻게(How)"는 Generator에게 맡긴다.
> 기획 단계에서 기술적 세부사항을 정의하면 오류가 전체로 전이된다.

### 1-1. 맥락 파악

병렬로 읽기:

- `CLAUDE.md` — 프로젝트 규칙 (Split File Contracts 포함)
- `docs/ai-context/current-task.md` — 진행 중 작업
- `git log --oneline -5` — 최근 변경

### 1-2. 완료 기준 작성

작업 요청을 분석하여 **구체적이고 검증 가능한 완료 기준** 3~7개를 작성한다.

완료 기준 규칙:

- "~되어야 한다" 형태의 명확한 문장
- **동작 검증**으로 확인 가능해야 함 (코드 리뷰가 아닌 실제 동작)
- 엣지 케이스 포함 (폴백, null, 빈 상태 등)
- Playwright로 자동 검증 가능하면 테스트 코드도 함께 계획

예시:

```
✅ 어드민에서 site_name 변경 시 카카오톡 공유 미리보기에 반영된다
✅ DB 값 없으면 site.config.json 폴백이 정상 작동한다
❌ Layout.astro를 수정한다 (이건 "어떻게"지 "무엇"이 아님)
```

### 1-3. 사용자 합의

완료 기준을 사용자에게 제시하고 **승인을 받은 후에만** Phase 2로 진행한다.
사용자가 수정/추가를 요청하면 반영한다.

## PHASE 2: GENERATOR (제작자)

> 완료 기준을 달성하기 위해 구현한다.
> 스프린트 단위로 작업하되, 각 스프린트는 검증된 기반 위에 쌓는다.

### 2-1. 스프린트 분할

완료 기준을 기반으로 구현 순서를 결정한다:

1. 기반이 되는 것 먼저 (데이터 모델, 유틸 함수)
2. 그 위에 쌓는 것 (UI, 연결)
3. 엣지 케이스/폴백 마지막

### 2-2. 구현

- 수정 전 반드시 파일 읽기
- Split File Contracts 참조 (동시 수정 필수 파일 확인)
- 핵심 파일 수정 시 PreToolUse hook이 규칙을 주입해줌 → 반드시 확인
- **외부 조언(외부 AI, 공식 문서, Stack Overflow)도 의심** — 이 프로젝트 컨텍스트에 맞는지 재검증

### 2-3. 정련 (Polishing) — 스프린트마다 필수

> 1차 작성은 초안이다. 읽고 줄이는 과정이 진짜 작성이다.
> "코드는 쓰는 게 아니라 지우는 것이다."

각 스프린트 코드 작성 후 **다음으로 넘어가기 전에** 자기 코드를 다시 읽는다:

1. 방금 쓴 파일 전체를 **Read 툴로 다시 읽는다** (기억에 의존 금지)
2. 각 줄에 질문:
   - "이 줄이 꼭 필요한가?"
   - "이 방어 코드는 실제 발생 가능한 케이스인가, 이론적 방어인가?"
   - "이 주석은 코드가 말하지 못하는 것만 설명하는가?"
   - "비슷한 일을 하는 기존 함수가 있어서 중복되지는 않는가?"
3. 제거할 게 있으면 삭제 → 다시 한 번 읽기
4. 2~3회 반복하고 **"더 이상 뺄 게 없을 때"** 완성
5. 제거된 줄 수 기록 (다음 스프린트 전 보고)

**경계선**: 정련은 기능 축소가 아니다. 완료 기준은 여전히 만족해야 한다.
가독성과 간결함을 동시에 추구한다.

### 2-4. 중간 빌드 검증

5개 파일 수정마다 `pnpm build` 실행. 깨졌으면 즉시 수정하고 다음으로 넘어가지 않는다.
검증된 기반 위에 쌓는다 — 모래 위에 16층을 올리지 않는다.

## PHASE 3: EVALUATOR (비판자) — 반드시 서브에이전트로 실행

> Generator(나)와 **별개의 관점**에서 결과물을 평가한다.
> **자기 평가 편향 차단**: Generator가 자기 코드를 평가하면 항상 "잘 됐다"고 판단한다.
> 따라서 Phase 3은 **반드시 별도 서브에이전트**로 실행한다. 예외 없음.

### 3-1. 빌드 검증

```bash
pnpm build
```

실패 시 → Phase 2로 돌아가 수정.

### 3-2. Playwright 스모크 테스트

```bash
pnpm test:e2e
```

실제 브라우저에서 페이지 로드, API 응답, OG 태그, 어드민 접근 등을 자동 검증한다.
**코드 리뷰가 아니라 실제 동작 확인.** 실험에서 Evaluator가 Playwright로 클릭/스크롤/스크린샷을 찍은 것과 같은 역할.

실패 시 → Phase 2로 돌아가 수정.

이번 작업에 특화된 Playwright 테스트가 필요하면 `e2e/` 디렉토리에 추가 작성한다.

### 3-3. 서브에이전트 코드 리뷰 (필수 — 스킵 불가)

**반드시** code-analyzer 또는 별도 Agent로 코드 리뷰를 실행한다.
Generator인 내가 직접 리뷰하지 않는다.

서브에이전트에게 전달할 정보:

1. 이번 작업의 완료 기준 목록
2. 변경된 파일 목록
3. "Generator의 자기 평가를 신뢰하지 말고 독립적으로 검증하라"는 지시
4. **"stale comment에 속지 마라"** — 주석과 실제 코드가 일치하는지 확인 요청

### 3-4. 완료 기준 대조

Phase 1에서 합의한 완료 기준을 **하나씩** 검증한다:

```
✅ 기준 1: [통과/실패] — 검증 방법 + 결과
✅ 기준 2: [통과/실패] — 검증 방법 + 결과
...
```

실패 항목이 있으면 → Phase 2로 돌아가 수정.

### 3-5. 시각 검증 (브랜치 작업 시 필수)

HIGH 위험도 작업은 Vercel 프리뷰 URL에서 **실제 눈으로 확인**:

- UI 변경: 모바일/데스크톱 뷰포트 각각
- 이미지 교체: 로고/아이콘처럼 **비율 제각각인 이미지**는 왜곡 여부 확인 (파트너 로고 넙대대 사례)
- 인터랙션: 클릭·스크롤·폼 제출

빌드 통과 ≠ 시각 정상. 둘 다 확인.

### 3-6. 관련 Guardrail 실행

변경한 파일을 분석하여 관련 guardrail을 실행:

| 변경 영역                                            | 실행할 guardrail   |
| ---------------------------------------------------- | ------------------ |
| priceEngine, calculate-price, create-order, PriceBox | `/price` 전체      |
| middleware, auth, upload, API 인증                   | `/audit` 전체      |
| CSS, 레이아웃, 모바일, 터치                          | `/design` 전체     |
| blockDefaults, ProductView, orderService             | `/health` 전체     |
| 여러 영역 교차                                       | `/check` 통합 실행 |

"핵심만" 돌리지 않는다. 관련되면 전체를 돌린다.

## PHASE 4: 반복 (Iteration)

### 실패 시 흐름

```
Phase 3 실패 → Phase 2 수정 → Phase 3 재평가
```

- 5회 반복 후에도 실패 → 사용자에게 **계속할지 판단 요청** (5회에서 자동 중단하지 않음)
- 실험에서 10번째 반복에서 창의적 도약이 일어남. 반복을 두려워하지 않는다.
- 같은 방식으로 3회 실패하면 → 땜질 대신 **근본 재설계** 고려
- 근본적 설계 문제 발견 시 → Phase 1로 돌아가 완료 기준 재협의

## PHASE 5: 완료 보고 + 자기 수정

### 5-1. 결과 보고

```markdown
## /work 결과

### 브랜치 정보

- 작업 브랜치: `refactor/{name}` (또는 main)
- PR URL: (브랜치 작업 시)

### 완료 기준 달성

- ✅ 기준 1: ...
- ✅ 기준 2: ...

### 변경 파일

| 파일 | 변경 내용 | 정련으로 제거한 줄 |
| ---- | --------- | ------------------ |

### 검증

- pnpm build: 통과/실패
- Playwright: N개 테스트 통과/실패
- 서브에이전트 리뷰: 결과 요약
- 시각 검증: 프리뷰 URL 확인 완료/미실시
- Guardrail: 실행 여부 + 결과

### 반복 횟수

- N회 반복 (실패 원인 + 수정 내용)
```

### 5-2. 상태 저장

- `docs/ai-context/current-task.md` 업데이트
- 미완료 항목 있으면 `docs/ai-context/checklist.md`에 추가
- 브랜치 작업이면 PR 생성

### 5-3. 자가 수정 (자가진화 핵심)

이번 세션에서:

- **이 도구(`work.md`)가 놓친 패턴**이 있었나? → 이 파일을 직접 수정
- **불필요해진 STEP**이 있나? → 제거
- **순서가 비효율적**이었나? → 재정렬
- 배운 교훈 → `docs/guardrails/shared/evolution-log.md`에 `[자동반성]`으로 기록

## 핵심 원칙 (편향 방지 체크리스트)

1. **완료 기준 없이 코딩 시작 금지** — Planner를 건너뛰지 않는다
2. **검증된 기반 위에 쌓기** — 빌드 깨진 채로 다음 작업 금지
3. **Evaluator는 반드시 서브에이전트** — 내가 내 코드를 평가하지 않는다
4. **Playwright로 실제 동작 검증** — 코드 리뷰만으로 "됐다"고 하지 않는다
5. **Guardrail 관련 영역 전체 실행** — "핵심만"이라고 범위를 줄이지 않는다
6. **반복을 두려워하지 않는다** — 5회에서 자동 중단하지 않고 사용자에게 물어본다
7. **같은 실패 3회 → 근본 재설계** — 땜질 반복하지 않는다
8. **과도한 리팩토링 금지** — 완료 기준에 없는 것은 안 건드린다
9. **정련 반복** — 1차 작성 → 읽기 → 줄이기. "더 뺄 게 없을 때"가 완성이다
10. **외부 조언도 의심** — 외부 AI, 공식 문서, Stack Overflow 답변 모두 출발점일 뿐 정답이 아니다
11. **위험한 작업은 브랜치에서** — HIGH 위험도는 main 직접 작업 금지, `refactor/{name}` 브랜치 + PR
12. **주석은 거짓말한다** — stale comment에 속지 마라. git blame으로 최신성 확인
13. **빌드 통과 ≠ 시각 정상** — UI 작업은 프리뷰 URL에서 눈으로 확인
14. **스스로를 개선한다** — PHASE 5-3에서 이 파일을 직접 수정한다
