## STEP 2: 점검 항목

### 2-1. 컴파일 (필수)

```bash
typst compile --root . --font-path fonts scripts/render-taepyeongcheonha-ch01.typ output/check.pdf
```

- 통과 = 트랙 A 회귀 없음
- 실패 = P0. 즉시 원인 파악 (폰트 경로, 스키마 변경, Typst 버전).

### 2-2. JSON 스키마 일치

- `content/*.json`이 D-006(스키마 v1) 모양인가
- `template.typ`이 디스패치하는 `blk.type`과 JSON의 `type` 값이 일치하는가
- 불일치 = P0.

### 2-3. PDF 육안 검증 체크리스트

`output/*.pdf`를 사용자가 열어보고:

- [ ] 챕터 시작 페이지에 쪽번호가 없는가
- [ ] 챕터는 홀수 페이지에서 시작하는가
- [ ] 첫 단락은 들여쓰기 없고, 둘째 단락부터 1글자 들여쓰기인가
- [ ] 양끝 정렬되어 있고, 한영 자간이 어색하지 않은가
- [ ] 한국어 금칙(`,` `.` `」` 가 행두에 가지 않는가) 준수되는가
- [ ] 쪽번호가 짝/홀수 페이지에서 바깥쪽에 위치하는가

미완 항목은 `docs/ai-context/checklist.md`에 기록.

### 2-4. 감사 보고서 미해결 항목

`docs/audits/`에서 가장 최근 보고서를 읽고:
- P0 위생 (H1~H3)이 처리되었는지
- 사용자가 결정 대기 중인 항목이 있는지
- 새로 발견된 함정이 있으면 보고서 갱신 또는 신규 작성

### 2-5. 핵심 원칙 위반 스캔

CLAUDE.md "핵심 원칙" 4개 위반 여부:
- [ ] 데이터 ↔ 스타일 분리 (JSON에 폰트·여백 박지 않았는지)
- [ ] AI 핵심 경로 미사용 (LLM API 호출 코드 0인지)
- [ ] 1단계 작업 규칙 (신국판 Classic 외 다른 조합 만들지 않았는지)
- [ ] 자유도 노출 (사용자가 폰트/자간/여백 직접 입력하는 UI 없는지)

## 핵심 원칙

1. **컴파일 통과가 1순위.** 깨진 빌드를 두고 다른 점검 의미 없음.
2. **PDF 육안 검증은 사용자가 한다.** Claude는 항목만 정리.
3. **트랙 B/C는 점검 대상 아님** — 코드가 없는 것을 점검하지 않는다.
4. **수정 전 반드시 파일 읽기.** 기억으로 수정하지 않는다.
5. **답이 명확한 것만 수정, 불명확한 것은 백로그.** 결정 박지 않는다.
