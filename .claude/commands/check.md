# /check — 통합 사이트 점검 (자가진화형)

> 4개 영역(보안·가격·디자인·상품/주문)을 한 번에 점검한다.
> Self-Evolving: 매 실행마다 배우고, 성장하고, 불필요한 것은 버린다.

## 정체성

너는 이 프로젝트의 **CTO이자 시니어 엔지니어이자 QA 리드**다.
4개 영역을 동시에 점검하고, 교차 이슈까지 찾아낸다.

## STEP 0: 진화 (절대 건너뛰지 않는다)

### 0-0. 로직 업데이트 (제일 먼저 — 필수)

**이 점검 도구 자체의 로직을 먼저 최신화한다.**

1. `CLAUDE.md` 읽기 → 새로 추가된 규칙, 변경된 아키텍처, Split File Contracts 변경 확인
2. `docs/ai-context/current-task.md` → 최근 어떤 작업이 이루어졌는지 파악
3. `git log --oneline -10` + `git diff --stat HEAD~5` → 최근 변경된 파일 목록으로 **이번 점검의 중점 영역** 결정
4. 새로 추가된 기능/블록/config가 있으면 → 해당 영역의 점검 항목에 반영

**왜 이게 먼저인가**: 점검 도구가 옛날 로직으로 돌면 새로 추가된 기능을 아예 모르고 지나감.

### 0-1. 지식 로드

병렬로 읽기:

- `docs/guardrails/shared/common-rules.md` → 공통 규칙

- `docs/guardrails/audit/rules.md` → 보안/품질 규칙
- `docs/guardrails/price/rules.md` → 가격 규칙
- `docs/guardrails/design/rules.md` → 디자인 규칙

- `docs/guardrails/check/history.md` → 이전 실행 결과

### 0-1b. /missed 피드백 우선 처리 (필수)

evolution-log.md에서 `[사용자발견]` + `[자동반성]` 항목을 찾아 이번 점검에서 최우선 검증.

### 0-3. 자기 정리

- 3회 이상 해당 없음 판정된 항목 → 제거
- 제거 시 보고

### 0-4. 메타 의심 (자가진화 핵심)

매 실행마다 이 파일 자체에 질문한다:

- 지난번 점검에서 **오진한 게 있나?** (evolution-log `[자동반성]` 확인)
- 최근 도입된 기능이 있는데 **점검 항목에 없는 게 있나?**
- 중복되거나 불필요해진 STEP이 있나? → 제거
- 순서가 비효율적인가? (예: 빌드 전에 코드 읽기, 수정 전에 의도 파악) → 재정렬
- 이 파일에 **쓸데없이 붙어있는 내용(로그 출력, 예전 에러 덤프)**이 있나? → 삭제

변경이 필요하면 이번 실행 끝에 STEP 5에서 이 파일을 직접 수정한다.

## STEP 1.5: 다각도 검토 (의도 파악 — 수정 전 필수)

> **코드 결함을 찾기 전에, 왜 이렇게 되어 있는지를 먼저 이해한다.**
> 의도를 모르고 수정하면 정상 동작을 깨뜨린다.

### 반드시 파악할 관계들

1. **블록 간 관계**: `linkedBlocks`로 연결된 블록들의 역할 (cover/inner/default). 한 블록 수정이 다른 블록에 미치는 영향
2. **4가지 블록 패턴 공존**: 현대 linkedBlocks / spring 통합 / 구형 inner_layer / 단층 flyer — 4개가 동시에 존재하는 이유 (하위호환)
3. **가격 3곳 동기화 의도**: ProductView(표시) / calculate-price(계산) / create-order(검증) — 같은 로직이 3곳에 있는 이유 (보안)
4. **절삭 3경로**: unitPrice / supply / payment — 각각의 비즈니스 의미와 적용 시점
5. **역할별 감지 폴백**: `getPaperBlockRole()` 4단계, `inferProductType()` 폴백 — 왜 여러 단계인지 (레거시 데이터 호환)
6. **숨은 추가금 3종**: guidePriceTotal / fileSpecPrice / designFee — priceEngine 밖에서 합산되는 이유

### 검토 프로토콜

수정 제안 전에 다음을 자문한다:

- "이 코드가 중복처럼 보이는데, 의도적 중복(보안/호환)은 아닌가?"
- "이 폴백이 불필요해 보이는데, 구형 DB 데이터가 이 경로를 타는 건 아닌가?"
- "이 하드코딩이 나쁜 패턴인데, 빌더 config로 바꾸면 기존 상품이 깨지지 않나?"
- "이 config 키를 바꾸면 Split File Contracts에 따라 몇 개 파일을 동시 수정해야 하나?"

**의도 불명확 시**: 수정하지 않고 P3(백로그)로 기록. "확인 필요: {의문점}"

## STEP 1.8: 실제 동작 검증 (Playwright)

> 코드 리뷰 전에 **실제 브라우저에서 동작을 확인**한다.
> Anthropic 하네스 실험의 핵심: Evaluator는 코드를 읽는 게 아니라 **실제로 클릭하고 스크린샷을 찍는다.**
> 코드 리뷰만으로 '됐다'고 하지 않는다.

### 실행

```bash
pnpm test:e2e
```

실패한 테스트는 스크린샷과 함께 `test-results/`에 저장된다.
실패 원인(locator mismatch, timeout, status code 등)을 확인 후 STEP 3 판정에 반영한다.

### 검증 항목

- 홈페이지 로드 + header/footer 존재
- OG 메타태그 (og:site_name, og:title, og:description) 존재
- 어드민 미인증 접근 → 로그인 리다이렉트
- calculate-price API → 양수 가격 반환
- Schema.org 구조화 데이터 존재

### 실패 시

Playwright 테스트 실패 항목은 **P0(즉시 수정)**으로 분류한다.
실제 동작이 안 되는 것은 코드 품질 이슈보다 우선한다.

## STEP 2: 전문가 진단 (4개 영역 병렬)

> **Agent 도구로 4개 영역을 병렬 실행한다.**
> **STEP 1.5의 다각도 검토를 에이전트에도 전달한다.**

### 영역 1: 보안·품질

**에이전트 1** — 다음을 점검:

- **깨진 것**: 빌드 에러, null 안전성, 데드 코드, Split File Contracts 동기화
- **위험한 것**: API 인증, XSS, 환경변수 노출, Supabase RLS, outsourced_config 노출
- **느린 것**: WebP 미적용, lazy loading 누락, srcset/sizes 없음, 번들 크기
- **아키텍처**: 코드 중복(DRY), 에러 처리, 폼 이중 제출 방지, 알림톡 fire-and-forget

### 영역 2: 가격

**에이전트 2** — 다음을 점검:

- **5경로 검증**: flyer, perfect, saddle, spring, outsourced 각 경로 0원/NaN 확인
- **추가금 3종 동기화**: guidePriceTotal, fileSpecPrice, designFee — 3곳 일치
- **product_type 안전성**: 5개 TEMPLATES 명시, inferProductType 로직, API 거부
- **outsourced 보안**: outsourced_config 클라이언트 미노출
- **절삭(applyRound)**: PriceBox / OrderSummary / create-order 3곳 동기화

### 영역 3: 디자인·UI

**에이전트 3** — 다음을 점검:

- **모바일 레이아웃**: overflow, 세로글자, flex/grid 깨짐 (375px 기준)
- **터치·인터랙션**: 44px 터치 타겟, 간격, 포커스 표시
- **시각적 일관성**: 토큰(--c-_, --sp-_, --fs-\*) 사용 vs 하드코딩 혼용
- **이미지**: aspect-ratio, object-fit, image_fit 시스템 정합성
- **이미지 주의**: 로고·아이콘처럼 **비율 제각각인 이미지**에 고정 width/height 부여 금지 (파트너 로고 넙대대 사례)
- **CSS 아키텍처**: @layer 위반, specificity, 미사용 CSS

### 영역 4: 상품·주문

**에이전트 4** — 다음을 점검:

- **DB 상품 스캔**: product-health API 호출 또는 코드 레벨 검증
- **가격 동기화**: ProductView / calculate-price / create-order 3곳 로직 일치
- **주문 플로우**: sessionStorage 키 일관성, priceInput/orderData 정합성
- **블록→가격 정합성**: extractDefaultsFromBlocks → customer 키 → 가격 경로
- **에러 처리**: API 실패 시 0원 표시, 사일런트 실패

## STEP 2.5: 다각도 검토 (오탐 필터링 — 수정 전 필수)

> STEP 2에서 수집된 모든 발견 사항을 **의도 파악 관점에서 재분석**한다.
> "코드가 이상해 보인다" ≠ "버그다". 의도된 구조를 버그로 잡으면 오히려 사고가 난다.

### 2.5-1. 의도적 구조 확인

각 발견 사항에 대해 다음을 자문:

- **"중복처럼 보이는데 의도적 중복(보안/호환)은 아닌가?"**
  - 가격 3곳 동기화 = 보안상 의도적 중복
  - applyRound 3곳 = 클라이언트/서버 각각 필요
  - 4가지 블록 패턴 공존 = 레거시 DB 데이터 호환

- **"폴백이 불필요해 보이는데 구형 데이터가 이 경로를 타는 건 아닌가?"**
  - `getPaperBlockRole()` 4단계 = 구형/신형 상품 모두 지원
  - `inferProductType()` = product_type null인 구형 상품 비상용

- **"하드코딩이 나쁜 패턴인데, 바꾸면 기존 상품이 깨지지 않나?"**
  - config 키 이름 변경 → Split File Contracts 전체 영향
  - DB에 저장된 값과 코드의 키가 일치해야 함

### 2.5-2. 영향 범위 확인

수정 제안 시 반드시:

1. **Split File Contracts 체크**: 이 수정이 몇 개 파일에 동시 영향을 주는지
2. **가격 경로 영향**: 5개 경로 중 어느 경로가 영향받는지
3. **DB 데이터 호환**: 이미 저장된 상품 데이터가 깨지지 않는지
4. **고객 화면 영향**: 수정이 고객이 보는 가격/UI에 영향을 주는지

### 2.5-3. 주석 의심 (stale comment 탐지)

> **주석은 거짓말을 한다.** 코드는 바뀌어도 주석은 안 바뀐다.
> 실제 사례: `ProductView.jsx:608-609` 주석이 이미 고쳐진 "FIXED_DELIVERY_OPTIONS 폴백 버그"를 여전히 경고해서 오진 유발.

발견 중 주석에 기대어 판단한 게 있다면:

- 실제 코드 동작을 다시 확인 (주석만 믿지 마라)
- git blame으로 해당 주석 작성 시점 + 이후 코드 변경 이력 확인
- "주석에 X라고 써있으니 X일 것이다" = 검증 아님

### 2.5-4. 오탐 분류

- **오탐 확인** → 발견 목록에서 제거, 이유 기록
- **의도 불명확** → P3(백로그)로 내리고 "확인 필요: {의문점}" 기록
- **진짜 버그 확인** → 판정 단계로 전달

## STEP 3: 판정 (4단계)

> STEP 2.5에서 오탐이 제거된 **확인된 이슈만** 분류한다.

| 분류                        | 기준                                                                                                                  | 행동                                    |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------- | --------------------------------------- |
| **P0 — 즉시 수정**          | 실제 동작이 깨진 것: Playwright 실패, 빌드 에러, 가격 0원, 보안 구멍, 고객 클레임 가능성                              | 이번 세션 내 수정                       |
| **P1 — 이번 세션 수정**     | 답이 명확한 버그·동기화 불일치·코드 정리                                                                              | 이번 세션 내 수정                       |
| **P2 — 제안 (사용자 판단)** | 개선 효과를 **수치로 증명** 가능한 것. "번들 -30%", "LCP -0.5초", "비용 월 -5만원" 등. "깔끔해진다"만으로는 자격 없음 | 수치와 함께 보고, 사용자 승인 후 진행   |
| **P3 — 백로그**             | 답이 없거나 지금 안 해도 되는 것                                                                                      | `docs/ai-context/checklist.md`에 기록만 |

### 판정 핵심 원칙

1. **"답이 있으면 수정, 없으면 기록"** — 해결책이 명확한 문제는 등급 따지지 말고 수정
2. **P2 제안은 근거 필수** — 체감이 아닌 수치로 효과를 증명할 수 있어야 제안 자격
3. **수정 루프 방지** — /check를 돌릴 때마다 끝없이 수정하지 않는다. 이번 작업과 직결된 것 + 답이 명확한 것만 수정

## STEP 3.5: 수정안 정련 (실행 전 필수)

> STEP 3에서 결정된 수정안을 **바로 실행하지 말고** 한 번 더 의심한다.
> 고치기 위해 새로 추가하는 코드가 또 다른 부채가 될 수 있다.

### 자문 체크리스트

각 수정안마다:

1. **더 작은 변경**으로 같은 효과를 낼 수 있는가?
2. **수정하려는 파일 전체를 방금 다시 읽었는가?** — 기억에 의존하지 마라. 구조가 바뀌었을 수 있다
3. 이 수정이 **다른 영역에 예기치 못한 영향**을 주지 않는가? (Split File Contracts 재확인)
4. 비슷한 버그를 **이미 고친 이력**이 있는가? (git blame + evolution-log)
5. 사용자가 **의도적으로 남긴 구조**는 아닌가? (STEP 1.5 다각도 검토 재확인)
6. 외부 AI/공식 문서가 권장하는 방법이 **이 프로젝트 규모·컨텍스트에 맞는가?** — 오버엔지니어링 경계

### 판정 수정

- **더 작게 가능** → 수정안 축소
- **영향 범위 큼** → P1 → P2 로 내리고 사용자 합의 후 진행
- **의도 불명확** → 수정 보류, P3 백로그로

### 수정안 자체의 정련

수정 코드를 쓴 **후**에도 한 번 더 읽고 줄인다:

1. 방금 쓴 파일을 Read 툴로 다시 읽기
2. 각 줄 질문: "이 줄이 꼭 필요한가?" "이 방어 코드는 실제 발생 가능한가?" "이 주석은 코드가 말하지 못하는 것만 설명하는가?"
3. 불필요한 부분 제거 → 다시 읽기
4. 2~3회 반복하고 "더 뺄 게 없을 때" 완성
5. 제거된 줄 수 기록

## STEP 4: 실행

1. **P0/P1 수정** — 답이 명확한 것은 즉시 코드 수정 (수정 전 반드시 파일 읽기)
2. **P2 제안 보고** — 수치/근거와 함께 사용자에게 제시, 사용자 판단 후 진행
3. **P3 기록** — `docs/ai-context/checklist.md`에 추가
4. **빌드 검증** — `pnpm build` 통과 확인
5. **위험도 높은 수정은 브랜치에서** — CdnImage 마이그레이션 같은 대규모 리팩토링은 `refactor/{name}` 브랜치에서 진행, PR로 병합

## STEP 5: 학습 + 자기 수정

1. 발견 패턴의 반복 가능성 평가
2. 반복 가능성 높은 패턴 → 해당 영역 rules.md에 추가
3. 빈 결과 항목 → 카운트 증가 (3회 누적 시 제거)
4. 교차 발견 → `docs/guardrails/shared/evolution-log.md`에 `[교차발견]` 기록
5. 실행 결과 → `docs/guardrails/check/history.md`에 기록
6. `docs/ai-context/current-task.md` 업데이트
7. **이번 세션에서 스스로 오진한 게 있으면** → evolution-log에 `[자동반성]` 기록
8. **이 파일(check.md) 자체에 개선할 점이 있으면 직접 수정** — STEP 0-4에서 찾은 메타 의심 결과 반영

## 핵심 원칙

1. **4개 영역 병렬 실행** — Agent로 동시에 돌려서 시간 단축
2. **교차 이슈 감지** — 단일 가드레일로는 못 잡는 영역 간 이슈 발견
3. **과도한 리팩토링 금지** — 실질적 이슈만 다룬다
4. **가격 계산은 성역** — 5개 경로 모두 검증
5. **수정 전 반드시 읽기** — 코드를 읽지 않고 수정하지 않는다
6. **STEP 0은 절대 건너뛰지 않는다** — 자가진화가 이 도구의 핵심
7. **스스로를 개선한다** — STEP 5-8에서 이 파일을 직접 수정한다
8. **Playwright 먼저** — 코드 리뷰 전에 실제 동작부터 확인한다
9. **자기 평가 편향 경계** — "괜찮아 보인다"를 신뢰하지 않는다. 도구로 검증한다
10. **범위 축소 금지** — "핵심만" "간단히"로 점검 범위를 줄이지 않는다
11. **정련 반복** — 수정안 1차 작성 → 읽기 → 줄이기. "더 뺄 게 없을 때"가 완성이다
12. **외부 조언도 의심** — 외부 AI, 공식 문서, Stack Overflow 답변 모두 출발점일 뿐 정답이 아니다. 이 프로젝트의 컨텍스트가 최우선
13. **주석은 거짓말한다** — stale comment에 속지 마라. 코드 동작을 직접 확인한다
14. **위험한 수정은 브랜치에서** — 대규모 리팩토링은 main에 직접 커밋 금지, `refactor/{name}` 브랜치 + PR
